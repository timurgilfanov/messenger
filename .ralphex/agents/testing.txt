Review test coverage and quality for this Messenger Android repo.

## Test Existence and Coverage

1. Missing tests
- New or changed logic should have tests in the owning module
- Error paths, validation branches, and state transitions should be covered

2. Correct test level
- JVM tests for domain/data logic where possible
- Android unit or `androidTest` only when Android framework or UI behavior requires it
- Interface/signature changes that impact `androidTest` sources should not leave those sources unvalidated

3. Integration-sensitive areas
- Repository/data-source mapping and error translation need targeted coverage
- ViewModel changes should cover state updates, side effects, and restore/retry behavior when relevant

## Messenger Testing Conventions

1. Prefer fakes and test fixtures over mocks/spies unless there is a strong reason not to
2. Use reproducible constants instead of current time or random values
3. Use `assertTrue` from `kotlin.test` instead of JVM `assert()`
4. Use Turbine cleanup intentionally; do not add `cancelAndIgnoreRemainingEvents()` by default
5. Keep tests behavior-focused rather than implementation-detail-focused

## What to Report

For each finding:
- Location: file and line reference
- Issue: what is missing or weak in the tests
- Impact: what bug or regression could slip through
- Fix: what test coverage or test change is needed

Report problems only - no positive observations.
